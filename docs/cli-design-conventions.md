# CLI Design Conventions

This document defines the design philosophy and conventions for the codemem CLI.
It exists so that all commands — whether written by a human or generated by an agent —
follow a predictable, composable shape that serves both interactive users and automation callers.

## Audiences

The CLI serves three audiences with different priorities:

| Audience | Primary concern | Example invocation |
|----------|----------------|-------------------|
| **Interactive user** | Discoverability, friendly output, helpful errors | `codemem search "auth flow"` |
| **Automation / scripts** | Stable JSON contracts, exit codes, no prompts | `codemem stats --json` |
| **Adapter / runtime plumbing** | Speed, minimal output, stdin/stdout pipes | `codemem claude-hook-ingest < event.json` |

When conventions conflict between audiences, prefer the automation caller.
Interactive niceties (color, spinners, prompts) should layer on top of a stable machine contract,
not the other way around.

## Command hierarchy

### Nesting depth

- Maximum two levels of subcommands: `codemem <group> <action>`.
- Three levels (e.g., `codemem sync coordinator list-devices`) is the hard ceiling and should be avoided for new commands. Existing three-level commands may stay until a migration replaces them.
- If a group would need a fourth level, promote the inner group to the top level instead.

### When to create a subcommand group

Create a group when:
- There are ≥3 related actions that share a noun (e.g., `memory show`, `memory forget`, `memory remember`).
- The actions share common flags or context (same DB, same identity, same remote).

Do **not** create a group for:
- A single action. A group with one subcommand is just indirection.
- Unrelated actions that happen to touch the same file. Group by user intent, not implementation.

### Grouping by concern

- **User-facing data operations** belong under their noun group: `memory show`, `memory export`, `memory import`, `db prune-memories`.
- **Operator/admin surfaces** that expose a distinct trust boundary or deployment concern belong in their own top-level group (e.g., `coordinator` for coordinator admin, not nested under `sync`).
- **Lifecycle/service management** belongs under the service noun: `serve start`, `serve stop`.
- **Plumbing/adapter commands** that are called by other programs (not humans) may stay top-level with clear naming: `claude-hook-ingest`, `mcp`.

### Naming

- Subcommand groups are **nouns**: `memory`, `sync`, `db`, `config`, `coordinator`.
- Actions are **verbs** or **verb-noun**: `search`, `show`, `enable`, `list-devices`, `prune-memories`.
- Prefer grouping related actions under their noun rather than top-level `verb-noun` compounds. If `export` and `import` both operate on memories, they belong under `memory export` and `memory import`, not as standalone `export-memories` / `import-memories`.
- Avoid abbreviations in command names. Flags may abbreviate (`--db` not `--database`) when the short form is universally understood.

## Arguments vs flags

### Positional arguments

Use a positional argument when **all** of these are true:
1. It is the **primary noun** the command acts on (the "what").
2. There is **exactly one** required positional, or at most two when the pair is unambiguous (e.g., `rename-project <old> <new>`).
3. The meaning is **obvious from context** without reading `--help`.

Good positional arguments:
```
codemem search <query>
codemem memory show <id>
codemem coordinator list-devices <group>
codemem memory import <file>
```

### Flags (options)

Use a flag when **any** of these are true:
1. The parameter is **optional** or has a default.
2. The parameter is **one of several** inputs (ambiguous ordering).
3. The parameter modifies **how** the command runs, not **what** it acts on.
4. The parameter is a **path, URL, or connection detail** (these are configuration, not subjects).

Good flags:
```
codemem search <query> --limit 10 --json
codemem sync enable --host 0.0.0.0 --port 7337
codemem serve start --config /path/to/config.json
```

### Required flags

Use `requiredOption` sparingly. If there is exactly one required subject, it should almost
always be a positional argument. Reserve required flags for the rare case where:
- A flag is mandatory but does not qualify as a positional (e.g., it names a secondary entity in a context where the command already takes a positional for the primary subject).
- The command would be meaningless or dangerous without it, but the value is not the command's primary subject.

**Anti-pattern to avoid:**
```
# Bad: --peer is the primary subject, should be positional
codemem sync bootstrap --peer <device-id>

# Better:
codemem sync bootstrap <peer-device-id>
```

### Boolean flags

- Use `--flag` for enable, not `--flag true`.
- Use `--no-flag` or a separate flag for disable. Prefer paired flags (`--enable-sync` / `--disable-sync`) when both directions are common.
- Never use `--flag=true|false` string booleans.

## Shared options

### Standard flags

These flags should be defined once via shared option-builder helpers and reused across commands.
Every standard flag has both a short form and a long form, defined consistently:

| Short | Long | Type | Description | Applies to |
|-------|------|------|-------------|-----------|
| `-d` | `--db-path <path>` | string | SQLite database path; overrides `CODEMEM_DB` | All commands that touch the store |
| `-c` | `--config <path>` | string | Config file path; overrides `CODEMEM_CONFIG` | Commands that read/write config |
| `-j` | `--json` | boolean | Emit structured JSON to stdout | All data-returning commands (see exemptions below) |

### Short flag conventions

- Every new flag should define both a short form and a long form.
- Short forms are single letters: `-j`, `-d`, `-c`, `-n`, `-l`, etc.
- Do not reuse the same short letter for different meanings across commands.
- Standard short forms defined in this doc take precedence over per-command inventions.
- Commander.js built-in `-h` (help) and `-V` (version) are reserved.

### Retired aliases

- `--db` is a legacy alias for `--db-path`. Accept silently but do not show in `--help`.
- `--user` and `--system` are Python-era Typer leftovers. Accept silently, ignore, do not show.

### Host/port disambiguation

`--host` and `--port` must not mean different things on different commands.
When a command controls a specific service, use an explicit prefix:

| Service | Flags |
|---------|-------|
| Viewer/serve | `--host`, `--port` (default meaning; this is the primary service) |
| Sync listener | `--sync-host`, `--sync-port` |
| Coordinator | `--coordinator-host`, `--coordinator-port` |

If only one service is involved and the command name makes it unambiguous, bare `--host` / `--port` are fine.

## Output contract

### stdout vs stderr

- **stdout**: the command's primary output. In `--json` mode, this is always valid JSON.
- **stderr**: progress messages, warnings, prompts, and human-formatted errors.
- Never mix JSON and non-JSON on stdout.

### `--json` mode

When `--json` is set:
- **Success**: print a single JSON object or array to stdout. Exit `0`.
- **Failure**: print a JSON object to stdout with at least `error` and `message` fields. Exit non-zero.
- **No output to stderr** except for fatal/unexpected crashes.

When `--json` is not set:
- Use `@clack/prompts` (`p.log.*`, `p.intro`, `p.outro`) for interactive feedback on stderr.
- Print primary data to stdout in a human-readable format.

#### `--json` examples

Success:
```json
{"device_id":"abc123","fingerprint":"SHA256:...","sync_reset":false}
```

Failure (usage error):
```json
{"error":"usage_error","message":"missing required argument: <group>"}
```

Failure (operation error):
```json
{"error":"grant_not_found","message":"Bootstrap grant not found: xyz789"}
```

#### `--json` exemptions

The following command classes do **not** need `--json`:

| Class | Reason | Examples |
|-------|--------|---------|
| **Lifecycle commands** | Side-effect only; success/failure is the exit code | `serve start`, `serve stop`, `serve restart` |
| **Stdin/stdout plumbing** | Already a structured pipe; adding `--json` would conflict with the transport | `claude-hook-ingest`, `enqueue-raw-event`, `mcp` |
| **Interactive setup** | Prompts are the primary UI; not intended for scripting | `setup` |

All other data-returning commands must support `--json`.

### Exit codes

| Code | Meaning |
|------|---------|
| `0` | Success |
| `1` | General failure (bad input, operation failed, resource not found) |
| `2` | Usage error (missing required arg, invalid flag combination) |

- Use `process.exitCode = N` (not `process.exit(N)`) to allow cleanup handlers to run.
- Never exit `0` on failure. Never exit non-zero on success.

### Error handling

- **Input validation errors** (bad flags, missing args): print a clear message and exit `2`. In `--json` mode, emit the structured failure object to stdout.
- **Operation errors** (resource not found, network failure): print context and exit `1`. In `--json` mode, emit the structured failure object to stdout.
- **Never** throw uncaught `Error` from a command action handler. Catch at the handler boundary, format, and exit. Uncaught throws produce ugly stack traces that confuse users and break automation parsers.

### Stdin/stdout plumbing commands

Commands that consume stdin (e.g., `claude-hook-ingest`, `enqueue-raw-event`):
- Must not prompt for input or confirmation.
- Should produce minimal, deterministic output on success (empty or a short status line).
- Should emit structured error output on failure (same `{error, message}` shape) to stdout.
- Exit codes follow the same convention as all other commands.

## Lifecycle commands

The viewer is the long-running runtime. Lifecycle management belongs under `serve`:

```
codemem serve start         # background by default
codemem serve stop
codemem serve restart
codemem serve start -f      # foreground; -f / --foreground
```

- Bare `codemem serve` (no action argument) runs in the **foreground**. This is the simple
  "I want to see what's happening" invocation. It is equivalent to `codemem serve start --foreground`.
- `codemem serve start` (with explicit action) runs in the **background** by default,
  which is the normal production/daemon mode.
- No legacy flags (`--stop`, `--restart`, `--background`). Use explicit action arguments.
- `sync start|stop|restart` should not exist as separate commands; they are lifecycle aliases for `serve` and create confusion about what process is being managed.

## Backwards compatibility

### Deprecation process

1. **Add** the new canonical command alongside the old one.
2. **Warn** on stderr when the old form is used: `Warning: 'codemem sync start' is deprecated, use 'codemem serve start'`.
3. **Hide** the old command from `--help` and shell completion after one release.
4. **Remove** the old command after at least two releases or one major version, whichever is longer.
5. **Never** silently change the behavior of an existing command name. If the semantics change, it needs a new name.

### Compatibility aliases

When a command is renamed or moved (e.g., `sync coordinator list-devices` → `coordinator list-devices`):
- Add a hidden compatibility alias at the old path that delegates to the new implementation.
- The alias must preserve the exact same behavior, arguments, flags, and output contract.
- The alias emits a deprecation warning on stderr, then runs the canonical command.
- Aliases do not appear in `--help` or shell completion.
- Aliases follow the same removal timeline as deprecated commands (2+ releases or 1 major version).

### Automation stability promise

- Any command with `--json` output is an automation contract. Its JSON shape may gain new fields but must not remove or rename existing fields without a major version bump.
- Positional argument order must not change.
- Exit codes must not change meaning.

## Documentation parity

- Every new command, renamed command, or flag change must update user-facing docs
  (`README.md`, `docs/user-guide.md`, and any relevant docs under `docs/`) in the same change.
- If a docs update is deferred, it must be tracked with an explicit issue linked to the implementation PR.
- Shell completion lists (`packages/cli/src/index.ts` completion handler) must stay in sync with the registered command tree.

## Style notes

- Prefer lowercase kebab-case for command and subcommand names: `list-devices`, not `listDevices` or `list_devices`.
- Prefer lowercase for flag values where case does not matter.
- Help text should be a short imperative phrase, not a sentence: `"database path"` not `"Specifies the path to the database."`.
- Descriptions in `--help` should fit in 80 columns including the flag column.

## Applying this to new commands

Before adding a new command, answer:
1. **Who is the primary audience?** (interactive / automation / adapter)
2. **What is the primary noun?** → That's your positional argument.
3. **What group does it belong to?** → If none, is it frequent enough for top-level?
4. **Does it need `--json`?** → Yes unless it falls in an exempt class (lifecycle, plumbing, interactive setup).
5. **What existing shared flags does it need?** → Use the option builders, don't copy-paste.
6. **Does it change user-facing behavior?** → Update docs in the same PR.
